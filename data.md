## Этап 7 — Уведомления и финальная сборка ✅

*Сшиваем всё в законченный end-to-end сценарий.*

- 7.1 ✅ NotificationsScreen на реальных данных (закрыто 2026-04-22, сессия A)
- 7.2 ✅ Push через Expo Notifications (закрыто 2026-04-22, сессия C — server-side OK, реальный push на iPhone отложен в 8.2)
- 7.3 ✅ Deep-link из пуша на нужный экран (закрыто 2026-04-22, сессия C)
- 7.4 ✅ Empty / loading / error по всем экранам (закрыто 2026-04-22, сессия B)

- 7.6 ✅ Инвайт в pool через ссылку (закрыто 2026-04-22, сессия A)
- 7.7 ✅ URL-парсер вишлист-item (закрыто 2026-04-22, сессия B — с известным ограничением)
- 7.8 ✅ Share-sheet для вишлиста (закрыто 2026-04-22, сессия A)

**Сессия A закрыта 2026-04-22.** Закрыты 7.1 + 7.6 + 7.8 параллельно через двух субагентов. Ключевые решения:
- Notifications: миграция-триггер `notify_on_friendship_accepted` (SECURITY DEFINER) — первый живой источник нотификаций; на клиенте INSERT в notifications запрещён RLS, пишут только триггеры/Edge Functions.
- Pool invite: без таблицы pool_invites (для MVP) — direct INSERT в pool_participants по клику на ссылку, маппинг ошибок 42501/23503/23505 → человеческие сообщения.
- Wishlist share: Share API на native + navigator.clipboard на web; `gifty.app/u/<handle>` — placeholder до Этапа 8.3.

Известный edge case (принят): для несуществующего pool_id Postgres RLS WITH CHECK срабатывает раньше FK — `pool_target(NULL)` возвращает NULL, политика падает с 42501. Битая ссылка показывает «это твой подарок» вместо «приглашение недействительно». Для MVP оставлено как есть.

**Сессия B закрыта 2026-04-22.** Закрыты 7.7 + 7.4 параллельно через двух субагентов (один пишет код, второй — аудит-only).

**7.7 URL-парсер:** первая Edge Function в проекте (`supabase/functions/parse-url`). Server-side fetch HTML → regex-парсер JSON-LD + Open Graph + Twitter cards + schema.org microdata. Браузерный User-Agent + Sec-Fetch хедеры. Клиентский wrapper `lib/parseItemUrl.ts`. Интеграция в `app/item-form.tsx`: pill-кнопка «Подтянуть» + автопарсинг на blur, заполняет ТОЛЬКО пустые поля.

Известное ограничение MVP: антибот-защита крупных магазинов (Yandex Market, WB, Ozon) возвращает 403 даже на полный браузерный набор хедеров — нужен headless browser или платный scraping API (отложено на Этап 9+). Для блокировок 403/429/498 — отдельное человеческое сообщение «Этот магазин блокирует автоматическое чтение. Заполни вручную». Также категорийные страницы магазинов вернут только site-wide og:title (название магазина) — пользователю нужна ссылка на конкретный товар.

**7.4 Empty/loading/error:** аудит показал, что 9 из 14 экранов уже соответствуют эталонному паттерну `wishlist.tsx`. Реальная правка нужна только в `event.tsx` — добавлен inline empty-state для пустого вишлиста героя (Gift иконка + «Пока пустой» + контекстная подсказка). `pool.tsx` уже был закрыт в 7.6 (loading/error/invitation states). `index.tsx` (dashboard loading) — отложено как средний приоритет, требует skeleton-UI а не глобального «Загружаю...» (он перекрыл бы hero-секцию).

**Сессия C закрыта 2026-04-22.** Закрыты 7.2 + 7.3 + 7.5. Push через Expo Notifications + deep-link + базовый a11y pass.

**7.2 + 7.3 Push + deep-link:** первая push-инфраструктура. `lib/usePushRegistration.ts` (request permission + регистрация Expo push token в `push_tokens` через upsert). `lib/notificationRouter.ts` (маппинг kind → router.push: friend_added → /friend-wishlist, pool_invite → /pool, и т.д.). Edge Function `send-push` берёт notification id, fetches push_tokens, отправляет batch через Expo Push API; чистит `DeviceNotRegistered` токены. Триггер `notifications_send_push` (AFTER INSERT ON notifications) вызывает `send-push` через `pg_net.http_post`. Server-side end-to-end smoke прошёл — pipeline работает (200, sent: 0 потому что токенов ещё нет).

Известное ограничение MVP: реальный push на **iPhone** через Expo Go SDK 53+ Apple отрубил — заработает только после dev build с EAS + Apple Developer ($99). Это **Этап 8.2**. На Android Expo Go — работает полностью. Для текущего теста (Константин на iPhone) пуши не доставляются, но инфраструктура готова и проверена на бэке.

**7.5 A11y базовый pass:** аудит показал ~75% покрытия (везде уже стоят accessibilityRole + accessibilityLabel). Применены **высокоприоритетные** правки: (а) `accessibilityRole="tablist"` на 7 tab-bar'ах + 2 wishlist-варианта (loading/empty), (б) `accessibilityHint="Формат: день, точка, месяц, точка, год"` на DateInput в event-form (пример паттерна для остальных форм). **Глубокая a11y-полировка** (text scaling Dynamic Type 1.5x+, accessibilityElementsHidden на декоративные иконки, accessibilityRole="status"/"alert" в loading/error блоках, тестирование с VoiceOver и TalkBack на устройствах) перенесена в **Этап 8.5** — это требует физических устройств для тестирования, не M-mode задача.

- 8.8 Соответствие 152-ФЗ — РКН-уведомление **после** 8.0 миграции, чек-лист (consent flag, anonymization, deletion endpoint)

**Решения, зафиксированные в Сессии D:**
- Архитектура: self-hosted Supabase на Yandex Cloud (research-артефакт `thoughts/research/2026-04-24-152-fz-compliance.md`, memory `project_gifty_152fz_decision.md`).
- Двухконтур (PII в РФ + Supabase для остального) **отвергнут** — user_id есть везде, кросс-БД RLS не работает, больше переделок и риск всё равно есть. Заметка опыта в Obsidian.
- Реестр ОРИ (мессенджеры) — **не идём**. Чат в pool позиционируется как закрытая координация участников сделки, не как общественные коммуникации. Отразить в Privacy Policy.
- Дистрибуция: Android first через RuStore (бесплатно, без $99); iOS через Apple Developer KZ-аккаунт (Kaspi-карта). Монетизация через продавцов (affiliate / комиссия) — Apple OK с этим, IAP не требуется. Memory `project_gifty_distribution.md`.
- Pools «скидываемся» сейчас без интегрированного шлюза — Apple OK. Если будем добавлять встроенные платежи — пересмотр.
- Бета (Этап 9): RuStore Internal Testing + APK через Telegram. TestFlight опционально (большинство тестеров на iPhone имеют российский Apple ID и не смогут установить).
