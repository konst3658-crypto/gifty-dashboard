## Этап 7 — Уведомления и финальная сборка 🔄

*Сшиваем всё в законченный end-to-end сценарий.*

- 7.1 ✅ NotificationsScreen на реальных данных (закрыто 2026-04-22, сессия A)
- 7.2 ⏳ Push через Expo Notifications
- 7.3 ⏳ Deep-link из пуша на нужный экран
- 7.4 ✅ Empty / loading / error по всем экранам (закрыто 2026-04-22, сессия B — частично, см. ниже)
- 7.5 ⏳ Accessibility pass (labels, readers, font scale)
- 7.6 ✅ Инвайт в pool через ссылку (закрыто 2026-04-22, сессия A)
- 7.7 ✅ URL-парсер вишлист-item (закрыто 2026-04-22, сессия B — с известным ограничением)
- 7.8 ✅ Share-sheet для вишлиста (закрыто 2026-04-22, сессия A)

**Сессия A закрыта 2026-04-22.** Закрыты 7.1 + 7.6 + 7.8 параллельно через двух субагентов. Ключевые решения:
- Notifications: миграция-триггер `notify_on_friendship_accepted` (SECURITY DEFINER) — первый живой источник нотификаций; на клиенте INSERT в notifications запрещён RLS, пишут только триггеры/Edge Functions.
- Pool invite: без таблицы pool_invites (для MVP) — direct INSERT в pool_participants по клику на ссылку, маппинг ошибок 42501/23503/23505 → человеческие сообщения.
- Wishlist share: Share API на native + navigator.clipboard на web; `gifty.app/u/<handle>` — placeholder до Этапа 8.3.

Известный edge case (принят): для несуществующего pool_id Postgres RLS WITH CHECK срабатывает раньше FK — `pool_target(NULL)` возвращает NULL, политика падает с 42501. Битая ссылка показывает «это твой подарок» вместо «приглашение недействительно». Для MVP оставлено как есть.

**Сессия B закрыта 2026-04-22.** Закрыты 7.7 + 7.4 параллельно через двух субагентов (один пишет код, второй — аудит-only).

**7.7 URL-парсер:** первая Edge Function в проекте (`supabase/functions/parse-url`). Server-side fetch HTML → regex-парсер JSON-LD + Open Graph + Twitter cards + schema.org microdata. Браузерный User-Agent + Sec-Fetch хедеры. Клиентский wrapper `lib/parseItemUrl.ts`. Интеграция в `app/item-form.tsx`: pill-кнопка «Подтянуть» + автопарсинг на blur, заполняет ТОЛЬКО пустые поля.

Известное ограничение MVP: антибот-защита крупных магазинов (Yandex Market, WB, Ozon) возвращает 403 даже на полный браузерный набор хедеров — нужен headless browser или платный scraping API (отложено на Этап 9+). Для блокировок 403/429/498 — отдельное человеческое сообщение «Этот магазин блокирует автоматическое чтение. Заполни вручную». Также категорийные страницы магазинов вернут только site-wide og:title (название магазина) — пользователю нужна ссылка на конкретный товар.

**7.4 Empty/loading/error:** аудит показал, что 9 из 14 экранов уже соответствуют эталонному паттерну `wishlist.tsx`. Реальная правка нужна только в `event.tsx` — добавлен inline empty-state для пустого вишлиста героя (Gift иконка + «Пока пустой» + контекстная подсказка). `pool.tsx` уже был закрыт в 7.6 (loading/error/invitation states). `index.tsx` (dashboard loading) — отложено как средний приоритет, требует skeleton-UI а не глобального «Загружаю...» (он перекрыл бы hero-секцию).
